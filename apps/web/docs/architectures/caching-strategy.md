# 문제 해결 방식 및 TanStack Query와의 비교

현재 블로그 프로젝트에서 캐싱 문제를 해결한 방식과, 질문하신 TanStack Query와의 차이점에 대해 설명해 드리겠습니다.

---

## 1. 현재 적용된 해결 방식 (`next/cache`)

이번 작업에서는 **Next.js의 서버 사이드 캐싱(Server-Side Caching)** 기능을 사용하여 문제를 해결했습니다.

*   **핵심 기술**: `unstable_cache` (Next.js App Router의 데이터 캐싱 API)
*   **작동 원리**:
    1.  Notion API를 호출하는 함수(`getPublishedPosts` 등)를 `unstable_cache`로 감쌌습니다.
    2.  **Revalidate(재검증) 시간**을 `21600초(6시간)`로 설정했습니다.
    3.  서버는 Notion에서 데이터를 한 번 가져오면, 이를 **서버 메모리나 파일 시스템에 저장**해둡니다.
    4.  이후 6시간 동안 들어오는 모든 유저의 요청에 대해 Notion API를 다시 호출하지 않고, **저장된 데이터를 즉시 반환**합니다.
*   **결과**:
    *   페이지 로딩 속도가 매우 빨라집니다 (HTML이 미리 만들어져 있거나 캐시된 데이터를 사용하므로).
    *   Notion API 호출 횟수를 획기적으로 줄여 API 속도 제한(Rate Limit)을 피할 수 있습니다.

---

## 2. TanStack Query (React Query) vs. Next.js Cache 비교

TanStack Query의 `staleTime` 설정과 현재 방식을 비교하면 다음과 같습니다.

| 특징 | **현재 방식 (Next.js Cache / ISR)** | **TanStack Query (React Query)** |
| :--- | :--- | :--- |
| **작동 위치** | **서버 (Server)** | **브라우저 (Client)** |
| **캐시 공유** | **모든 유저가 캐시를 공유함** | 각 유저(브라우저)마다 별도로 캐시됨 |
| **SEO (검색엔진)** | **최상 (HTML에 데이터가 포함됨)** | 설정에 따라 다름 (클라이언트 페칭 시 SEO 불리) |
| **초기 로딩** | **매우 빠름 (완성된 HTML 전송)** | 로딩 스피너가 보일 수 있음 |
| **API 호출량** | **매우 적음** (6시간에 1번만 호출) | 유저 수만큼 호출될 수 있음 |
| **주 용도** | 블로그, 뉴스, 마케팅 페이지 (정적 콘텐츠) | 대시보드, SNS, 관리자 페이지 (동적/실시간 데이터) |

## 3. 결론: 무엇이 더 좋은가?

**이 블로그 프로젝트에는 현재 적용한 `Next.js Cache` 방식이 훨씬 더 좋습니다.**

이유는 다음과 같습니다:
1.  **블로그의 특성**: 블로그 글은 자주 바뀌지 않습니다. 6시간마다 한 번만 갱신되어도 충분합니다.
2.  **SEO (검색 엔진 최적화)**: 블로그는 구글 검색에 잘 노출되는 것이 중요합니다. Next.js 캐싱을 사용하면 서버에서 완성된 HTML을 보내주므로 검색 로봇이 내용을 완벽하게 읽을 수 있습니다.
3.  **Notion API 보호**: TanStack Query를 사용하여 클라이언트에서 데이터를 가져오게 되면, 방문자가 100명이면 Notion API 호출도 100번 발생합니다. 반면, 서버 캐싱은 방문자가 100명이든 1만 명이든 6시간 동안은 **단 1번**만 호출합니다.

**요약하자면:**
TanStack Query는 "유저와의 상호작용이 많은 앱(예: 좋아요, 댓글 실시간 반영)"에 좋고, **현재와 같은 콘텐츠 중심의 블로그에는 Next.js의 서버 캐싱이 성능과 비용 면에서 압도적으로 유리합니다.**
